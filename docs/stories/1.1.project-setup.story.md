### Architecture Context
[Source: architecture/high-level-architecture.md#technical-summary]
- **Architecture Pattern**: Rust fullstack monolithic architecture deployed as single binary PWA
- **Web Framework**: Axum 0.8+ for HTTP server and routing
- **Template Engine**: Askama 0.14+ for server-side HTML rendering
- **Event System**: Evento 1.1+ for internal event-driven architecture
- **Database**: SQLite 3.40+ for embedded data persistence

[Source: architecture/high-level-architecture.md#repository-structure]
- **Structure**: Single Rust workspace monorepo with separate crates for modularity
- **Tool**: Cargo workspaces (native Rust solution)
- **Organization**: Functional separation through Rust crates while maintaining single binary output

### Data Models
[Source: architecture/data-models.md#user]
Core entities that will need database schema:
- **User**: uuid id, email, password_hash, name, family_size, dietary_restrictions, cooking_skill_level, timestamps
- **Recipe**: uuid id, title, description, timing, difficulty, servings, ingredients array, instructions array
- **MealPlan**: uuid id, user_id, week_start_date, meals array, status enum
- **ShoppingList**: uuid id, meal_plan_id, items array, sharing configuration

### API Specifications
[Source: architecture/api-specification.md#rest-api-specification]
Foundation endpoints to prepare for:
- `/api/auth/register` and `/api/auth/login` for user authentication
- `/api/meal-plans` for meal planning functionality
- `/api/recipes` for recipe management
- `/api/shopping-lists/{mealPlanId}` for shopping lists

### Tech Stack Requirements
[Source: architecture/tech-stack.md#technology-stack-table]
**Required Dependencies with Versions:**
- Rust 1.70+
- Axum 0.8+ (HTTP server, routing)
- Askama 0.14+ (server-side HTML templates)
- SQLx 0.8+ (async database, no compile-time query checking)

- Evento 1.1+ (event-driven architecture)
- rust-i18n (internationalization)
- config 0.15+ (configuration management)
- twinspark-js (progressive client-side interactivity)

**Additional Dev Tools:**

### Testing Requirements
**Testing Framework Setup:**
- Unit tests in `tests` modules within each crate
- Integration tests in `/tests` directory  
- Use `tokio-test` for async testing
- Use `rstest` for parameterized tests

**Specific Test Scenarios Required:**
1. **Health Endpoint Tests:**
   - Test `/health` returns 200 OK with proper JSON structure
   - Test health response includes version, database status, uptime fields
   - Test health endpoint responds within 100ms under normal conditions
   - Test health endpoint behavior when database is unavailable

2. **Configuration Loading Tests:**
   - Test loading from environment variables
   - Test loading from TOML/YAML configuration files  
   - Test configuration defaults when no config files present
   - Test configuration validation for required fields

3. **Database Initialization Tests:**
   - Test SQLite database creation and migration execution
   - Test database connection establishment
   - Test migration rollback functionality (if implemented)

4. **Server Startup Tests:**
   - Test server starts and binds to configured port
   - Test server graceful shutdown
   - Test hot reload functionality works correctly

**Performance Requirements:**
- Health endpoint must respond within 100ms
- Server startup must complete within 5 seconds
- Database initialization must complete within 2 seconds

### Development Scripts and Error Handling
**Recommended Development Scripts to Create:**
- `scripts/dev-setup.sh` - Complete development environment initialization
- `scripts/reset-db.sh` - Reset database with fresh migrations
- `scripts/health-check.sh` - Test health endpoint during development

**Error Scenarios to Handle:**
1. **Health Endpoint Error Responses:**
   - Return 503 Service Unavailable if database connection fails
   - Return 500 Internal Server Error for configuration issues
   - Include error details in response for development environment only

2. **Database Connection Errors:**
   - Graceful startup failure with clear error message if database unavailable
   - Connection retry logic with exponential backoff
   - Migration failure handling with rollback capability

3. **Configuration Errors:**
   - Clear validation errors for missing required configuration
   - Helpful error messages pointing to configuration documentation
   - Default value warnings in development mode
