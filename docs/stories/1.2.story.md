# Story 1.2: User Authentication System

## Status
Done

## Story
**As a** prospective user,  
**I want** to create an account and securely log into the platform,  
**so that** I can access personalized recipe management and meal planning features.

## Acceptance Criteria
1. User registration form with email, password, and basic profile information
2. Secure password hashing using industry-standard algorithms
3. Login form with email/password authentication
4. JWT-based session management with secure token storage
5. Password reset functionality via email verification
6. Email verification for new account activation
7. Logout functionality that invalidates session tokens
8. Basic user profile page showing account information
9. Form validation with clear error messages for invalid inputs
10. Protection against common security vulnerabilities (CSRF, brute force, etc.)

## Dev Notes

### Previous Story Insights
From Story 1.1: Infrastructure is complete with JWT secret validation (32+ character requirement), configuration management through `src/config/` modules, and production-ready setup with comprehensive testing framework.

### User Data Model Requirements
**User Schema**: UUID primary key, email (unique), password_hash, dietary_preferences (TEXT[]), skill_level (enum), household_size (integer), kitchen_equipment (JSONB), timestamps  
**Password Security**: Use argon2 for hashing with industry-standard parameters  
**Session Management**: JWT tokens with Redis storage for refresh tokens and rate limiting  
*[Source: architecture/data-models.md#user, architecture/database-schema.md#users-table]*

### Authentication Architecture  
**Service Layer**: Authentication service in `src/services/auth.rs` with register_user, authenticate_user, validate_token functions  
**Middleware**: JWT validation middleware in `src/middleware/auth.rs` for protected routes  
**Repository**: User repository in `src/repositories/user_repo.rs` with CRUD operations  
**Token Strategy**: JWT access tokens (1 hour) + Redis refresh tokens (30 days)  
*[Source: architecture/components.md#authentication-service, architecture/backend-architecture.md#authentication-and-authorization]*

### Database Schema Details
**Users Table**: CREATE TABLE with UUID primary key, unique email constraint, password_hash VARCHAR(255), dietary_preferences TEXT[], skill_level CHECK constraint, kitchen_equipment JSONB  
**Indexes**: Email uniqueness enforced, created_at/updated_at for audit trail  
**Migration**: Create as `migrations/002_create_users_table.sql` following existing pattern  
*[Source: architecture/database-schema.md#users-table]*

### File Structure Requirements
**Routes**: `src/routes/auth.rs` for authentication endpoints  
**Templates**: `templates/pages/auth/` directory with login.html, register.html layout  
**Middleware**: `src/middleware/auth.rs` for JWT validation  
**Models**: `src/models/user.rs` for User struct definition  
**Services**: `src/services/auth.rs` for business logic  
*[Source: architecture/unified-project-structure.md#project-structure]*

### Security Requirements
**Password Hashing**: argon2 with secure parameters, never store plaintext  
**JWT Handling**: Use jsonwebtoken crate, validate tokens in middleware  
**CSRF Protection**: Implement CSRF tokens for state-changing operations  
**Rate Limiting**: Use Redis for tracking login attempts and rate limiting  
**Session Security**: Secure token storage, proper logout invalidation  
*[Source: architecture/components.md#authentication-service, architecture/coding-standards.md#critical-fullstack-rules]*

### Frontend Integration
**Templates**: Use Askama templating with built-in escaping, no raw HTML  
**Forms**: Alpine.js for client-side validation and reactivity  
**Layout**: Separate auth layout template for authentication pages  
**Validation**: Both client-side immediate feedback and server-side validation  
*[Source: architecture/tech-stack.md#technology-stack-table, architecture/coding-standards.md#template-security]*

### Configuration Integration
**Environment Variables**: Access through config module, never directly from std::env  
**JWT Secret**: Use existing 32+ character validation from Story 1.1  
**Database**: Use existing PgPool configuration from `src/config/database.rs`  
**Redis**: Leverage existing Redis configuration for session storage  
*[Source: architecture/coding-standards.md#environment-variables]*
