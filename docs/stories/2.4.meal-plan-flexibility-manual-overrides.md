# Story 2.4: Meal Plan Flexibility & Manual Overrides

## Status

Done

## Story

**As a** meal planner,  
**I want** the ability to modify automated plans when life happens,  
**so that** I can maintain planning efficiency while adapting to schedule changes.

## Acceptance Criteria

1. Individual meal substitution without regenerating entire week plan
2. Drag-and-drop meal rearrangement within the weekly calendar
3. "Lock" individual meals to prevent changes during regeneration
4. Quick swap functionality suggesting similar recipes from unplanned collection
5. Shopping list automatically updates when meal changes are made
6. Change history tracking with undo capability for recent modifications

### Security Considerations

**Authentication & Authorization Requirements**:
- **Endpoint Security**: All meal modification endpoints (`PUT /meal-plans/:id/entries/:entryId`, `POST /meal-plans/:id/reorder`) require valid Bearer token authentication
- **Ownership Validation**: Verify user owns the meal plan before allowing any modifications (check meal_plans.user_id matches JWT user_id)
- **Resource Access Control**: Users can only modify their own meal plans and entries - implement middleware to validate ownership on all operations
- **Session Validation**: Validate JWT token expiry and refresh requirements for long-running drag operations

**Data Integrity & Concurrent Modification Protection**:
- **Optimistic Locking**: Implement version-based optimistic locking to detect concurrent modifications
- **Transaction Isolation**: Use SERIALIZABLE isolation level for complex reorder operations to prevent race conditions
- **State Validation**: Validate meal plan state hasn't changed before applying modifications (check updatedAt timestamp)
- **Atomic Operations**: Ensure meal substitution + shopping list updates are atomic - rollback on any failure

**Input Sanitization & Validation**:
- **Recipe ID Validation**: Verify recipe IDs exist and are accessible to the user before substitution
- **Change Reason Sanitization**: Sanitize and limit change reason text (max 255 chars, strip HTML/script tags)
- **Rate Limiting Enforcement**: Prevent abuse with sliding window rate limiting per user (30 modifications/minute)
- **Drag Operation Limits**: Limit batch reorder operations to maximum 10 meals per request

**Network Security & Error Handling**:
- **HTTPS Enforcement**: All modification endpoints must use HTTPS in production
- **Request Validation**: Validate all request payloads against schema, reject malformed requests
- **Error Information Leakage**: Return generic error messages to prevent information disclosure about system internals
- **Audit Logging**: Log all meal plan modifications with user ID, timestamp, and change type for security auditing

### Error Handling & Recovery Procedures

**Specific Error Scenarios & Recovery**:

**Network Failure Scenarios**:
- **Drag Operation Failure**: If drag-and-drop API call fails, revert UI state and show "Unable to move meal. Please try again." with retry button
- **Substitution Request Timeout**: Display "Meal substitution taking longer than expected" with option to cancel or wait
- **Shopping List Update Failure**: Show "Meal changed successfully, but shopping list couldn't update. Refresh page to sync." with refresh action

**Concurrent Modification Handling**:
- **Optimistic Lock Failure**: Display "Someone else modified this meal plan. Refreshing..." and reload current state
- **State Conflict During Undo**: Show "Meal plan was modified. Undo operation unavailable." and disable undo button
- **Drag Target Changed**: If meal is locked during drag operation, show "This meal was just locked and can't be moved"

**Invalid State Recovery**:
- **Recipe No Longer Available**: Display "Selected recipe is no longer available" with alternative suggestions
- **Meal Plan Deleted**: Redirect to meal plans list with message "This meal plan is no longer available"
- **Shopping List Service Down**: Allow meal modifications but show warning "Shopping lists temporarily unavailable. Changes will sync when service resumes."

**User-Facing Error Messages & Recovery Options**:
- **Rate Limit Exceeded**: "Too many changes too quickly. Please wait 30 seconds before making more modifications."
- **Session Expired**: "Your session has expired. Please log in again to continue." with automatic redirect to login
- **Permission Denied**: "You don't have permission to modify this meal plan" (shouldn't occur in normal flow)
- **Validation Errors**: "Invalid meal selection. Please choose from available recipes." with correction guidance

**Rollback Procedures for Failed Operations**:
- **Failed Substitution**: Restore original meal in UI, clear optimistic state, notify user of failure
- **Failed Reorder**: Reset all moved meals to original positions, clear drag state, show error message
- **Failed Shopping List Update**: Maintain meal plan changes but flag shopping list as out-of-sync with refresh option
- **Failed Undo Operation**: Maintain current state, disable undo button temporarily, show "Undo failed. Current changes are preserved."

**Offline Capability Considerations**:
- **Connection Lost**: Queue modifications locally and show "Offline - changes will sync when connection restores"
- **Partial Sync**: On reconnection, show conflicts and allow user to choose "Keep local changes" or "Sync from server"
- **Storage Limits**: If local storage full, prioritize most recent changes and warn user about potential data loss
