# Story 3.1: Recipe Rating & Review System

## Status

🚀 **PRODUCTION READY** - QA PASSED with 95/100 quality score

## Story

**As a** community member,  
**I want** to rate and review recipes from the community database,  
**so that** I can help others discover quality recipes and avoid disappointing meals.

## Acceptance Criteria

1. 5-star rating system with optional written reviews for recipes
2. Community recipe database separate from personal collections with aggregated ratings
3. Rating distribution display (number of 1-star, 2-star, etc. ratings)
4. Review moderation system flagging inappropriate content
5. Personal rating history tracking user's contributed reviews
6. Recipe recommendation engine prioritizing highly-rated community recipes

### Data Models & Relationships
**Recipe Rating Model** [Source: architecture/data-models.md#RecipeRating, architecture/9-database-schema.md]:
- `id`: UUID primary identifier
- `recipeId`: UUID reference to rated recipe 
- `userId`: UUID reference to rating contributor
- `rating`: integer 1-5 star rating scale
- `review`: optional text feedback (max length validation required)
- `difficulty`: enum assessment vs original complexity
- `wouldCookAgain`: boolean recommendation indicator
- Database constraints: unique constraint on (recipeId, userId) to prevent duplicate ratings
- Automatic rating aggregation via triggers updating recipes.average_rating and recipes.total_ratings

**Recipe Model Extensions** [Source: architecture/data-models.md#Recipe]:
- Add `isPublic`: boolean for community visibility
- Add `isCommunity`: boolean to separate community vs personal recipes
- Existing `averageRating` and rating distribution fields already defined in schema
- Rating aggregation handled by `update_recipe_rating_aggregates()` function

### API Specifications
**Rating Endpoints** [Source: architecture/api-specification.md, architecture/11-backend-architecture.md]:
- `POST /api/v1/recipes/:id/rating`: Submit recipe rating with JWT auth required
- `GET /api/v1/recipes/:id/ratings`: Paginated rating list with review text
- `GET /api/v1/community/recipes`: Browse community recipes with rating sorting
- `GET /api/v1/users/me/ratings`: Personal rating history endpoint
- All endpoints require Bearer token authentication via middleware
- Rate limiting on rating submission: 5 ratings per minute per user
- Response caching for community recipe lists (1 hour TTL)

### Frontend Component Architecture
**Component Structure** [Source: architecture/10-frontend-architecture.md, architecture/6-components.md]:
- Use Atomic Design pattern: RatingStars (atom) → ReviewCard (molecule) → RatingsList (organism)
- State management via Redux Toolkit with optimistic updates for rating submission
- Error boundary handling for rating service failures
- Accessibility support: screen reader labels, keyboard navigation, high contrast mode
- Performance: Lazy loading for rating lists, image optimization for community recipes

### Technical Constraints
**Performance Requirements**:
- Rating submission API: <500ms response time
- Community recipe loading: <2 seconds for 50 recipes
- Rating aggregation: Real-time updates via database triggers
- Review content moderation: <200ms processing time

**Business Rules**:
- One rating per user per recipe (enforced by unique constraint)
- Rating scale: 1-5 stars (integer validation)
- Review text: Optional, max 500 characters, profanity filtering required
- Community recipes: Minimum 3 ratings before appearing in recommendations
- Rating history: Permanent record for analytics (soft delete only)

### Error Handling & Recovery Procedures
**Specific Error Scenarios & Recovery**:

**Rating Submission Failures**:
- **Duplicate Rating**: Return 409 Conflict with message "You have already rated this recipe. Update your existing rating instead."
- **Invalid Rating Value**: Return 400 Bad Request with "Rating must be between 1 and 5 stars"  
- **Recipe Not Found**: Return 404 with "Recipe not available for rating"
- **Authentication Required**: Return 401 with redirect to login flow

**Content Moderation Failures**:
- **Inappropriate Content Detected**: Return 400 with "Review contains inappropriate content. Please revise."
- **Moderation Service Down**: Accept review with pending status, manual review queue
- **False Positive Filtering**: Provide appeal mechanism in user rating history

**Community Recipe Loading Errors**:
- **Rating Service Unavailable**: Display recipes without rating information, show cached ratings when available
- **Database Performance Issues**: Implement circuit breaker, fallback to basic recipe list
- **Empty Rating Results**: Show helpful message "Be the first to rate this recipe!"

**User-Facing Error Messages & Recovery Options**:
- **Network Errors**: "Unable to submit rating. Tap to retry." with exponential backoff
- **Server Overload**: "High traffic detected. Your rating will be processed shortly." with queue position
- **Validation Errors**: Inline form validation with specific field error messages

### Security Considerations
**Rating System Security** [Source: architecture/15-security-and-performance.md]:
- JWT token validation for all rating endpoints
- Rate limiting: Maximum 5 ratings per minute per user
- Input sanitization for review text to prevent XSS attacks
- SQL injection prevention using parameterized queries
- Content Security Policy headers for review display
- Audit logging for all rating modifications and deletions

### Caching Strategy
**Performance Optimization** [Source: architecture/11-backend-architecture.md]:
- Community recipe lists cached for 1 hour in Redis
- Individual recipe ratings cached for 30 minutes
- Rating aggregation results cached until new rating submitted
- User rating history cached for 15 minutes
- Cache invalidation on rating updates using Redis pipelines
