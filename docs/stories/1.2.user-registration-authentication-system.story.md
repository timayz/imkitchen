# Story 1.2: User Registration and Authentication System

## Status
Approved

## Story
**As a** potential user,
**I want** to create an account and securely log in,
**so that** I can access personalized meal planning features.

## Acceptance Criteria

1. **User Crate:** Create imkitchen-user crate with User aggregate, validated value objects (Email, Password with validator derive macros)
2. **Sync Input Validation:** Email format, password strength validation using validator 0.20+ directly in Axum handlers
3. **Database Operations:** Direct SQLx database queries for email existence checks and user validation
4. **User Registration:** Simple user creation with password hashing and email verification workflow
5. **Authentication Service:** Direct login validation with secure session management and cookie handling
6. **Database Queries:** Direct SQLx queries for user lookup, session validation, and profile management
7. **CLI Integration:** imkitchen binary uses imkitchen-web library to start server with dependency on imkitchen-user for authentication
8. **Askama + Tailwind Templates:** LoginForm.html, RegistrationForm.html, UserDashboard.html with Tailwind utility classes and type-safe binding to user domain
9. **TwinSpark Sync Validation:** Login form `ts-req` → direct Axum validation → immediate Askama error fragments
10. **TwinSpark Async Validation:** Email availability check `ts-req` → direct database query → Askama response fragments
11. **JavaScript-Free Forms:** Registration and login forms use TwinSpark with server-side validation patterns
12. **CLI Server Management:** `imkitchen web start` command initializes and runs the web server from imkitchen-web library
13. **Email Verification:** User registration triggers email verification workflow using notification service
14. **TDD Validation Testing:** Write tests first for both sync validation (direct) and async validation (database) patterns
15. **Session Management:** Secure session handling with cookie-based authentication and validation
16. **Database Views:** User data models and queries implemented with direct SQLx operations
17. **Crate Boundaries:** Clear separation between user domain logic (user crate) and presentation logic (web crate)

### User Domain Model Requirements
[Source: architecture/data-models.md]
- **User Profile Model Structure:**
  - user_id: Uuid - Unique identifier for user account
  - email: Email - Validated email address for authentication using validator derive macros
  - dietary_restrictions: Vec<DietaryRestriction> - List of dietary constraints
  - family_size: FamilySize - Number of people (1-8) for meal planning with range validation
  - cooking_skill_level: SkillLevel - Beginner/Intermediate/Advanced for recipe complexity
  - available_cooking_time: CookingTime - Weekday/weekend time availability

- **SQLx CRUD Implementation:**
  ```rust
  impl UserProfile {
      async fn register_user(pool: &SqlitePool, user_data: CreateUserData) -> Result<User, AppError>
      async fn update_profile(pool: &SqlitePool, user_id: Uuid, profile_data: UpdateProfileData) -> Result<(), AppError>
      async fn change_dietary_restrictions(pool: &SqlitePool, user_id: Uuid, restrictions: Vec<DietaryRestriction>) -> Result<(), AppError>
  }
  ```

### Database Schema Requirements
[Source: architecture/database-schema.md]
- **User Context Schema (SQLite):**
  ```sql
  CREATE TABLE user_profiles (
      user_id TEXT PRIMARY KEY,
      email TEXT UNIQUE NOT NULL,
      password_hash TEXT NOT NULL,
      family_size INTEGER CHECK(family_size BETWEEN 1 AND 8),
      cooking_skill_level TEXT CHECK(cooking_skill_level IN ('Beginner', 'Intermediate', 'Advanced')),
      weekday_cooking_minutes INTEGER DEFAULT 30,
      weekend_cooking_minutes INTEGER DEFAULT 60,
      created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
      updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
  );
  ```

- **Dietary Restrictions (Normalized):**
  ```sql
  CREATE TABLE dietary_restrictions (
      user_id TEXT REFERENCES user_profiles(user_id),
      restriction_type TEXT NOT NULL,
      severity TEXT DEFAULT 'Strict',
      PRIMARY KEY (user_id, restriction_type)
  );
  ```

[Source: architecture/coding-standards.md]
- **Type Safety Across Boundaries:** All data models must be defined in shared crate with validation

- **TwinSpark Patterns:** All dynamic interactions must use ts-* attributes, no custom JavaScript
- **Error Handling:** All handlers must return Result<T, AppError> with proper error conversion
- **Validation First:** Input validation with validator before any business logic execution
- **Testing Required:** Minimum 90% test coverage, TDD for all new features
- **Crate Boundaries:** Domain crates cannot depend on web crate, only shared types
