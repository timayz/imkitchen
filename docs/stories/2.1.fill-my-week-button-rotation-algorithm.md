# Story 2.1: "Fill My Week" Button & Rotation Algorithm

## Status

Done

## Story

**As a** meal planning user,
**I want** one-tap automated weekly meal generation,
**so that** I can eliminate planning decision fatigue while ensuring recipe variety.

## Acceptance Criteria

1. Prominent "Fill My Week" button on main dashboard automatically populates entire weekly calendar
2. Rotation algorithm ensures no recipe repeats until full collection has been used
3. Intelligent distribution across meal types (breakfast, lunch, dinner) with variety optimization
4. Algorithm considers prep time constraints and complexity distribution throughout week
5. Generation completes within 2 seconds with visual loading indicators
6. Users can regenerate plans if unsatisfied with initial selection

### Core Workflow Implementation

**"Fill My Week" Generation Workflow** [Source: architecture/8-core-workflows.md#primary-workflow-fill-my-week-meal-plan-generation]

- **Step 1**: User authentication and preference retrieval from cache
- **Step 2**: Recipe pool generation with dietary restriction filtering
- **Step 3**: Intelligent meal plan algorithm with variety optimization
- **Performance**: Complete workflow must execute in <2 seconds with Redis caching
- **Algorithm Priorities**: 1) Dietary compliance, 2) Variety across week, 3) Skill level appropriateness, 4) Prep time distribution, 5) Ingredient overlap optimization

**Recipe Pool Algorithm** [Source: architecture/8-core-workflows.md]

- Primary: Cached recipes by user preferences (100+ recipes for optimal selection)
- Fallback: External API integration with search filtering
- Pool size: Minimum 150 recipes requested for optimal filtering and variety

### Testing

**Testing Standards for Algorithm Story** [Source: architecture/16-testing-strategy.md]

**Test File Locations**:

- Backend: `apps/backend/tests/rotation_algorithm_test.go`, `apps/backend/tests/meal_generation_performance_test.go`
- Integration: `apps/backend/tests/fill_my_week_workflow_test.go`
- Frontend: `apps/mobile/__tests__/FillMyWeekButton.test.tsx`

**Algorithm Testing Strategy**:

- **Unit Tests**: Rotation algorithm edge cases, variety optimization, complexity distribution
- **Integration Tests**: Full generation workflow with database operations
- **Algorithm Validation**: Recipe pool sufficiency, dietary restriction compliance, rotation accuracy

**Performance Testing Requirements**:

- **Primary Requirement**: <2 second generation validation with K6 load testing
- **Edge Case Scenarios**:
  - Insufficient recipe pool handling (< 150 recipes): Should fallback to external API with graceful degradation
  - Concurrent user generation requests: Up to 10 simultaneous users without performance degradation
  - Cache failure scenarios: Redis unavailable should maintain <5 second generation time via database fallback
  - Database connection issues: Connection pool exhaustion and recovery testing
  - Algorithm timeout handling: Generation exceeding 2 seconds should return partial results with user notification
- **Load Testing Scenarios**:
  - Baseline: Single user generation under optimal conditions (<2s target)
  - Stress: 50 concurrent users with realistic recipe pools (95th percentile <3s)
  - Spike: Sudden traffic bursts (100 users in 10 seconds)
  - Endurance: Sustained load over 30 minutes to detect memory leaks
  - Recipe pool variation: Testing with 50, 150, 500, 1000+ recipe pools

**Test Coverage Requirements**:

- Algorithm logic: 90%+ unit test coverage
- Integration workflow: Complete end-to-end test coverage
- Performance: Load testing with realistic recipe pool sizes and user concurrency
- Error scenarios: All failure modes and recovery mechanisms tested

### Technical Constraints

**Performance Requirements** [Source: architecture/tech-stack.md, architecture/high-level-architecture.md]

- **Generation Time**: <2 seconds end-to-end (hard requirement from PRD)
- **Caching Strategy**: Redis caching for recipe pools, user preferences, and generated plans
- **Database Optimization**: PostgreSQL query optimization with proper indexing for recipe selection
- **Algorithm Efficiency**: In-memory processing with minimal database calls during generation

**Rotation Algorithm Constraints**

- **Memory Management**: Rotation state persistence in JSONB fields for scalability
- **Concurrency**: Thread-safe rotation tracking for concurrent user requests
- **Data Consistency**: ACID compliance for rotation state updates

### Security and Performance Considerations

**Authentication and Rate Limiting** [Source: Previous stories implementation]

- JWT token validation required for all meal plan generation requests
- Rate limiting: 5 requests per minute for generation endpoint to prevent abuse
- User context isolation to ensure rotation tracking per-user accuracy

**Caching and Performance** [Source: architecture/11-backend-architecture.md#high-performance-caching-layer]

- Multi-level Redis caching: User preferences, recipe pools, generated plans
- Cache invalidation strategy for recipe updates and user preference changes
- Fail-open caching design to maintain service availability during Redis outages
